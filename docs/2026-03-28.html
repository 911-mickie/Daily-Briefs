💡 CONCEPT REFRESHER

Mixture of Experts (MoE)

In an MoE model, the feed-forward layer (or sometimes the full transformer block) is replaced by a set of N expert networks, with a learned router that selects a sparse subset — typically top-2 — of experts to process each token. Only the selected experts run during a given forward pass, meaning total model parameters can scale without proportionally scaling compute per token. This is why models like Mixtral 8x7B have ~47B total parameters but only ~13B active per token. In practice, MoE introduces load-balancing challenges: without an auxiliary loss term forcing even expert utilization, a few experts get overtrained and the rest collapse. The common misconception is that MoE simply speeds up inference — in reality, serving MoE models requires all experts resident in memory, making memory bandwidth the bottleneck, not FLOPS.

Q1: Why does MoE reduce compute per token without reducing total model capacity? → Only the top-K routed experts execute per token; the rest of the parameters exist in memory but don't contribute to that forward pass.
Q2: What is the load-balancing problem in MoE and how is it typically addressed? → Without regularization, the router collapses to always selecting the same experts; an auxiliary load-balancing loss penalizes uneven expert selection frequency during training.
Q3: How does MoE interact with tensor parallelism in multi-GPU inference, and what's the key communication overhead? → Expert parallelism routes tokens across GPUs where different experts live, requiring All-to-All collectives per MoE layer — a fundamentally different communication pattern than tensor parallelism's All-Reduce.

• NOT VALID vs. NOT ENFORCED Constraints in PostgreSQL
What's new: PostgreSQL 18 adds NOT ENFORCED constraint state (from SQL:2023), distinct from the existing NOT VALID. NOT VALID is a temporary migration state — it skips historical data validation but enforces on new rows immediately, with the expectation you'll run VALIDATE CONSTRAINT later. NOT ENFORCED is permanent documentation-only: PostgreSQL will never enforce it, making it useful for query planner hints and schema documentation without runtime overhead. Currently scoped to CHECK and FOREIGN KEY constraints.
Practical takeaway: Use NOT VALID for safe zero-downtime constraint additions on large tables; use NOT ENFORCED only when you need the planner to see a constraint it can exploit for optimization without paying enforcement cost.
