• <b><a href="https://huggingface.co/papers/2603.27481">LLaVA-DyMoE: Dynamic MoE with Drift-Aware Token Assignment for Continual Learning of VLMs</a></b>
This paper tackles catastrophic forgetting in multimodal continual instruction tuning — specifically the failure mode where old-task tokens get routed to newly added experts in dynamic MoE architectures, causing forgetting despite expert isolation. The authors call this "routing drift" and frame it as a "Token's Dilemma": ambiguous and old tokens in new-task data provide minimal learning value but corrupt old-expert behavior when misrouted. LLaVA-DyMoE introduces drift-aware token assignment that steers problematic tokens away from new experts, with expert-group separation via targeted regularization — adding zero inference overhead. This is directly applicable to production VLM systems that need to absorb new domains without full retraining.
<i>Interview angle:</i> Be ready to explain how MoE routing decisions interact with continual learning objectives and why freezing existing experts alone is insufficient to prevent forgetting.

• <b><a href="https://huggingface.co/papers/2603.28610">ResAdapt: Adaptive Resolution for Efficient Multimodal Reasoning in Long-Video MLLMs</a></b>
ResAdapt addresses the computational wall of processing high-resolution video in MLLMs: instead of uniform token compression post-encoding (which loses detail), it learns input-side resolution budgets per frame before encoding, leaving the backbone untouched. A lightweight Allocator trained with Cost-Aware Policy Optimization (CAPO) dynamically concentrates pixel budget on information-dense frames and aggressively compresses redundant ones. The result is support for up to 16x more frames under the same visual token budget with a >15% gain on reasoning-intensive benchmarks. For production video-document or long-form video pipelines, this is a near drop-in efficiency upgrade.
<i>Interview angle:</i> Interviewers will probe why input-side adaptation outperforms output-side compression and how you'd train the Allocator without ground-truth frame importance labels.

Today's concept: Metadata Filtering in Vector Search

<b>Metadata Filtering in Vector Search</b>
Metadata filtering allows retrieval systems to constrain the candidate set before or during vector similarity search using structured attributes — locale, document type, user role, date range — rather than relying solely on embedding proximity. In production RAG systems, pure semantic similarity is insufficient when the same query should return different documents for different tenants, regions, or access levels. Pre-filtering (filtering the index before ANN search) reduces recall risk from over-restriction, while post-filtering (filtering after ANN) risks returning too few results if the filter is selective; hybrid approaches (filtering during index traversal) balance both. A common misconception is that metadata filtering is just a WHERE clause bolted onto vector search — in reality, filter cardinality, index structure (HNSW vs. IVF), and filter pushdown support vary significantly across vector stores and materially affect latency and recall.

<b>Q1:</b> What's the difference between pre-filtering and post-filtering in vector search, and when does each fail? → <i>Pre-filtering restricts the search space before ANN, risking recall loss on small filtered sets; post-filtering applies after ANN, risking too few valid results when the filter is highly selective.</i>

<b>Q2:</b> Ring uses metadata-driven filtering for locale-specific content — what metadata fields would you index, and how would you prevent cross-locale leakage? → <i>Index locale code, content region, and language tag as filterable fields; enforce mandatory filter injection at query construction time so no request reaches the retriever without locale constraints applied.</i>

<b>Q3:</b> How does metadata filter selectivity interact with HNSW graph traversal, and what happens to recall as the filter becomes more restrictive? → <i>Highly selective filters cause HNSW to explore large portions of the graph before finding enough valid candidates, degrading both latency and recall — mitigated by filtered HNSW implementations that skip invalid nodes during traversal or by partitioning the index per filter value when cardinality is low.</i>

• <b><a href="https://postgr.es/p/7ws">The Hidden Behavior of plan_cache_mode in PostgreSQL</a></b>
<i>What's new:</i> PostgreSQL's default <i>plan_cache_mode = auto</i> uses custom plans (actual parameter values) for the first five executions of a prepared statement, then switches permanently to a generic plan if its estimated cost is equal or lower. This silent switch can cause dramatic plan changes mid-session on skewed data distributions — the post demonstrates this with a pgbench schema on Postgres 18. The options are <i>force_custom_plan</i> (always re-plan, higher overhead) and <i>force_generic_plan</i> (never re-plan, risks bad plans on skewed params).
<i>Practical takeaway:</i> If you're seeing inexplicable query slowdowns after the sixth execution of a prepared statement, check your plan cache mode and inspect the generic plan with <i>EXPLAIN</i> using representative parameter distributions.

• <b><a href="https://planetscale.com/blog/high-memory-usage-in-postgres-is-good-actually">High Memory Usage in Postgres Is Good, Actually</a></b>
<i>What's new:</i> This piece clarifies a common ops misread: high RAM utilization in Postgres (e.g., 80% shown in a dashboard) typically reflects the OS page cache keeping hot data close to the CPU, not memory pressure or OOM risk. Unlike sustained high CPU — which directly degrades query latency — high memory utilization alone doesn't signal degraded performance. The distinction matters especially when shared_buffers and OS cache together absorb your working set.
<i>Practical takeaway:</i> Don't right-size Postgres instances down based on memory utilization percentage alone — monitor cache hit ratios and disk read rates instead to determine whether RAM is actually a bottleneck.
