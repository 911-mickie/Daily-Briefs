• <b><a href="https://huggingface.co/papers/2603.25158">Trace2Skill: Distill Trajectory-Local Lessons into Transferable Agent Skills</a></b>
The core problem is that automated skill generation for LLM agents either overfits to individual trajectories or relies on shallow parametric knowledge, creating brittle, non-transferable skills. Trace2Skill deploys a parallel fleet of sub-agents to holistically analyze a diverse pool of execution trajectories, extracts lessons, and uses inductive reasoning to hierarchically consolidate them into a unified, conflict-free skill directory — no parameter updates, no episodic retrieval at inference time. Key result: declarative skills evolved by a 35B model (Qwen3.5-35B) improved a 122B agent by up to 57.65 absolute percentage points on WikiTableQuestions, with strong cross-model and out-of-distribution transfer. This is directly relevant to any team building reusable tool-use or reasoning skill libraries for production agents.
<i>Interview angle:</i> How do you design skill libraries for LLM agents that generalize across task distributions without requiring per-task fine-tuning?

• <b><a href="http://bair.berkeley.edu/blog/2025/11/01/rl-without-td-learning/">RL without TD learning — divide and conquer for off-policy RL</a></b>
TD learning (Q-learning) has a fundamental scalability problem: value estimation errors compound through Bellman backups across long horizons, making it brittle for complex, long-horizon tasks. This BAIR post introduces a divide-and-conquer RL paradigm as an alternative — decomposing long-horizon problems into subproblems rather than propagating value estimates step-by-step. The approach works in the off-policy setting, which is critical for domains like robotics and dialogue where data collection is expensive and you can't afford to discard historical experience. As of 2025, on-policy RL (PPO, GRPO) has solid scaling recipes; this work targets the open problem of scaling off-policy RL to complex tasks.
<i>Interview angle:</i> Why does TD error compounding make Q-learning brittle for long-horizon tasks, and what are the theoretical alternatives?

• <b><a href="https://huggingface.co/papers/2603.25730">PackForcing: Short Video Training Suffices for Long Video Sampling</a></b>
Autoregressive video diffusion models suffer from linear KV-cache growth, temporal repetition, and compounding errors during long-video generation. PackForcing introduces a three-partition KV-cache strategy — sink tokens (full-res anchor frames), mid tokens (32x spatiotemporal compression via dual-branch 3D conv + low-res VAE re-encoding), and recent tokens (full-res for local coherence) — with dynamic top-k selection and Temporal RoPE Adjustment to handle dropped token position gaps. Result: coherent 2-minute 832x480 videos at 16 FPS on a single H200, bounded at 4 GB KV cache, with 24x temporal extrapolation (5s training → 120s generation), trained on only 5-second clips. The hierarchical compression strategy maps directly to long-context memory management problems in any autoregressive system.
<i>Interview angle:</i> How do you design KV-cache compression strategies that preserve global coherence while bounding memory footprint in long-context generation?

• <b><a href="https://huggingface.co/papers/2603.25716">Hybrid Memory for Dynamic Video World Models</a></b>
Existing video world model memory mechanisms treat environments as static canvases, failing when dynamic subjects exit and re-enter the frame — they produce frozen, distorted, or vanished subjects. The paper introduces HyDRA, a memory architecture that compresses context into tokens and uses spatiotemporal relevance-driven retrieval, treating the model simultaneously as a precise archivist for static backgrounds and a motion tracker for dynamic subjects. They also release HM-World, a 59K-clip dataset with decoupled camera/subject trajectories across 17 scenes and 49 subjects, purpose-built for evaluating hybrid coherence. The retrieval architecture — selectively attending to relevant motion cues rather than full context — is a direct analogue to selective memory retrieval in long-running agentic systems.
<i>Interview angle:</i> How do you design retrieval mechanisms that distinguish between stable context (background) and dynamic, time-sensitive state in a unified memory architecture?

<b>Knowledge Distillation</b>

Knowledge distillation transfers learned representations from a large "teacher" model to a smaller "student" model, with the student trained not just on ground-truth labels but on the teacher's output distribution — soft probability vectors (logits) that carry richer signal than one-hot labels because they encode the teacher's confidence and inter-class relationships. The intuition is that a teacher saying "60% cat, 35% lynx, 5% dog" teaches more than a label saying "cat." In production, this is how teams get frontier-model capability into edge-deployable or cost-efficient inference — exactly what Apple is doing with Gemini, and what Trace2Skill does across model scales. A common misconception is that distillation only applies to final output logits; in practice, feature-level distillation (matching intermediate layer activations) and reasoning-trace distillation (training on chain-of-thought outputs) often outperform logit-only approaches, especially for complex reasoning tasks.

<b>Q1:</b> Why are soft labels from a teacher model more informative than hard one-hot labels for training a student? → <i>Soft labels encode the teacher's full probability distribution, revealing similarity structure between classes that one-hot labels discard entirely.</i>
<b>Q2:</b> What is temperature scaling in distillation, and what does raising the temperature do to the teacher's output distribution? → <i>Temperature T divides the logits before softmax; higher T produces a softer, more uniform distribution that exposes smaller probability differences and makes the inter-class signal stronger for the student.</i>
<b>Q3:</b> When would reasoning-trace distillation (training on chain-of-thought outputs) outperform standard logit distillation for a code generation student model? → <i>When the task requires multi-step planning or intermediate state tracking, the reasoning traces carry causal structure that output logits alone cannot encode, giving the student a richer supervision signal for the generalization pattern rather than just the final answer.</i>

• <b><a href="https://postgr.es/p/7wo">PostgreSQL 19: COPY TO now supports JSON/NDJSON format</a></b>
<i>What's new:</i> A new <i>JSON</i> format option for <i>COPY TO</i> exports query results or table data as newline-delimited JSON (NDJSON), one object per line. It supports column list filtering, emitting only named columns per JSON object. The format is currently <i>COPY TO</i> only (not <i>COPY FROM</i>) and is incompatible with standard text/CSV options including <i>HEADER</i>, <i>DELIMITER</i>, <i>NULL</i>, <i>FORCE QUOTE</i>, and <i>FORCE NOT NULL</i>.
<i>Practical takeaway:</i> Use <i>COPY TO ... (FORMAT JSON)</i> in PostgreSQL 19 to directly stream NDJSON from the database for downstream ingestion pipelines, eliminating the transform step that currently requires application-layer serialization.

• <b><a href="https://postgr.es/p/7wp">Good CTE, Bad CTE — PostgreSQL planner behavior since PG12</a></b>
<i>What's new:</i> Prior to PostgreSQL 12, CTEs acted as hard optimization fences — the planner materialized them fully and could not push predicates in or leverage indexes on underlying tables. Since PG12, CTEs are inlined, materialized, or handled in-between depending on how they're written, giving the planner room to optimize but introducing behavior that differs by query structure in ways developers frequently don't anticipate.
<i>Practical takeaway:</i> If you're using CTEs to "force execution order," audit whether your queries actually need <i>MATERIALIZED</i> hints or whether CTE inlining (the PG12+ default) is silently changing your query plan in ways that break expected performance.

• <b><a href="https://blog.bytebytego.com/p/ep208-load-balancer-vs-api-gateway">Load Balancer vs API Gateway — system design distinction</a></b>
<i>What's new:</i> Crisp breakdown of the architectural boundary: load balancers have a single job (distribute traffic across backend instances at L4/L7), while API gateways handle cross-cutting concerns — auth, rate limiting, request transformation, routing to microservices, and observability. Conflating them in architecture leads to either overloading your LB with logic it shouldn't own or deploying an API gateway where a simple LB would suffice.
<i>Practical takeaway:</i> In microservice deployments, use a load balancer for traffic distribution at the infrastructure layer and an API gateway for service-level policy enforcement — don't bolt auth and rate limiting onto your LB just because it can technically handle it.
