💡 CONCEPT REFRESHER

Retrieval-Augmented Generation (RAG)

RAG grounds a language model's responses by retrieving relevant documents at inference time rather than baking all knowledge into model weights. The retrieval step typically uses dense vector search (e.g. FAISS, pgvector) over chunked document embeddings, then injects the top-k results into the prompt context. The core trade-off is retrieval precision vs latency — more chunks mean better coverage but longer prompts and slower generation. A common misconception is that better embeddings always fix a RAG system; often the bottleneck is chunking strategy or reranking, not the embedding model itself.

What is the difference between sparse and dense retrieval in RAG?

Sparse (BM25) matches keywords exactly; dense retrieval uses semantic embeddings and finds conceptually similar content even with different wording.

How would you handle a RAG system that retrieves irrelevant chunks?

Add a cross-encoder reranker after initial retrieval to score relevance more accurately before passing chunks to the LLM.

What are the failure modes of naive chunking strategies?

Fixed-size chunks split mid-sentence or mid-concept, breaking context; prefer semantic or paragraph-aware chunking with overlap.
