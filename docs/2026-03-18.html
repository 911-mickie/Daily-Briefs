• Subagents — Agentic Engineering Patterns (Simon Willison)
Claude Code's subagent pattern dispatches a fresh model instance with a new context window to handle scoped subtasks — like repo exploration — returning only a structured summary to the parent agent. This preserves the top-level context budget while allowing arbitrarily large codebases to be explored without blowing the main window. The key insight is that subagents aren't just parallelism — they're a context management strategy: the parent never touches raw file contents, only processed summaries. This maps directly to how Lexi handles codebase ingestion: batch summarization is essentially a subagent pattern where each summary is a compressed, context-efficient representation of raw file content.
Interview angle: Expect questions on how you decompose large agentic tasks without exceeding context limits — the subagent/hierarchical delegation pattern is the canonical answer.

Today's concept: Context Window Constraints and Subagent Delegation

Context Window Constraints and Subagent Delegation
LLMs have a hard upper bound on tokens they can process in a single forward pass — currently around 1M tokens at the frontier, but with measurable quality degradation above ~200K tokens due to attention dilution and "lost in the middle" effects. Context is not free: filling a 1M token window incurs quadratic attention cost during prefill and holds proportional KV cache memory during decode. The subagent pattern solves this by decomposing tasks hierarchically — dispatching fresh model instances with scoped prompts, then returning only structured summaries upward, keeping the parent agent's context lean and high-signal. The critical misconception is that "more context = better results" — empirically, selective context curation outperforms raw context stuffing, which is why RAG over long documents beats shoving entire documents into the prompt.

Q1: Why do benchmarks show better quality below 200K tokens even when models support 1M? → Attention scores get diluted over long sequences, causing models to underweight relevant tokens far from the query — the "lost in the middle" phenomenon.
Q2: In a RAG system, when would you choose subagent delegation over retrieval? → When the subtask requires multi-step reasoning or exploration over a structured artifact (like a codebase), not just lookup — retrieval returns chunks, subagents return conclusions.
Q3: How does the KV cache interact with context window size during decode, and what's the memory implication? → KV cache grows linearly with sequence length and must be held in GPU HBM throughout decoding — a 1M token context at 70B scale can consume hundreds of GBs, making decode memory-bandwidth-bound and constraining batch size.
