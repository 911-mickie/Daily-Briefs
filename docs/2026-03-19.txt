• <b><a href="https://simonwillison.net/2026/Mar/18/llm-in-a-flash/#atom-everything">Autoresearching Apple's "LLM in a Flash" to run Qwen 397B locally</a></b>
Dan Woods applied Apple's "LLM in a Flash" technique — streaming model weights from SSD into DRAM on demand rather than holding them resident — to run Qwen3.5-397B-A17B (209GB on disk, 120GB quantized) at 5.5+ tokens/second on a 48GB MacBook Pro M3 Max. The approach exploits the MoE property that each token only activates a subset of experts, so only those expert weights need to be paged in per forward pass. Expert weights were quantized to 2-bit while non-expert components (embeddings, routing matrices) remained at full precision at 5.5GB resident in memory, and the system dropped active experts per token from 10 to 4 to reduce SSD I/O. This is a concrete, reproducible demonstration that flash-tier storage can substitute for DRAM at inference time for massive MoE models — relevant for on-device and edge deployment scenarios.
<i>Interview angle:</i> An interviewer might probe the memory hierarchy tradeoffs in LLM inference — DRAM vs. NVMe bandwidth constraints and how MoE sparsity changes the calculus.

• <b><a href="https://jack-clark.net/2026/03/16/importai-449-llms-training-other-llms-72b-distributed-training-run-computer-vision-is-harder-than-generative-text/">PostTrainBench: Can LLMs autonomously fine-tune other LLMs?</a></b>
PostTrainBench (from University of Tübingen, MPI, and Thoughtful Lab) measures whether frontier coding agents — Claude Code, Codex CLI, Gemini CLI — can autonomously build a training pipeline from scratch, choose data sources and methods, and improve a base model against a target benchmark, all within 10 hours on a single H100. • <b><a href="https://lilianweng.github.io/posts/2024-07-07-hallucination/">Extrinsic Hallucinations in LLMs</a></b>
Lilian Weng draws a precise boundary between in-context hallucination (output inconsistent with provided context) and extrinsic hallucination (output ungrounded by world knowledge encoded during pre-training), arguing the latter is under-studied despite being the more fundamental reliability problem. Root causes span pre-training data quality issues (stale, incorrect, or missing web-crawled content) and fine-tuning pathologies where SFT and RLHF can introduce or amplify fabrication. The post outlines mitigation strategies including retrieval augmentation, factuality-targeted RLHF, and model self-calibration for uncertainty acknowledgment. For RAG system designers, this framing is directly actionable: knowing which hallucination type you're fighting determines whether retrieval, better grounding prompts, or training-time fixes are the right lever.
<i>Interview angle:</i> Expect a question on how RAG addresses in-context vs. extrinsic hallucination differently, and what RAG cannot fix.

Today's concept: Parallelism Strategies in LLM Inference

<b>Parallelism in LLM Inference</b>
Large models can't fit on a single GPU and single-GPU throughput is often insufficient for production SLAs, so inference systems decompose computation across devices using multiple parallelism strategies simultaneously. <b>Tensor parallelism</b> shards individual weight matrices across GPUs — each GPU holds a slice of each layer and results are reduced via AllReduce after each operation, keeping all GPUs active but requiring low-latency intra-node interconnects. <b>Pipeline parallelism</b> assigns full layers to different GPUs in sequence, reducing communication volume but introducing pipeline bubbles during prefill. <b>Context parallelism</b> distributes the sequence dimension across GPUs so that long-context prefill (which is compute-bound and scales quadratically with sequence length) can be parallelized; GPUs must share partial attention outputs via AllGather. <b>Expert parallelism</b> applies specifically to Mixture-of-Experts models, routing each token's active experts to the GPU that holds those expert weights — requiring AllToAll collectives for token dispatch and collection. In practice, production systems like Meta's combine all four strategies with different configurations for prefill vs. decoding stages, since prefill is compute-bound and decoding is memory-bandwidth-bound.

<b>Q1:</b> Why does decoding require a different optimization strategy than prefill in LLM inference? → <i>Decoding generates one token at a time with small compute but must read the full KV cache and model weights each step, making it memory-bandwidth-bound rather than compute-bound.</i>

<b>Q2:</b> What communication primitive does expert parallelism in MoE models require, and why is it more expensive than AllReduce? → <i>AllToAll, because each token must be routed to a specific expert on a potentially different GPU — the data volume and routing irregularity make it harder to pipeline than the symmetric AllReduce used in tensor parallelism.</i>

<b>Q3:</b> If you're serving a 70B dense model with tensor parallelism across 8 GPUs and latency is still too high, what would you investigate first and why? → <i>AllReduce communication overhead between GPUs — at 8-way tensor parallelism, every layer requires a synchronization barrier, and if inter-GPU bandwidth is saturated or latency is high (e.g., PCIe vs. NVLink), the communication cost dominates compute time.</i>
