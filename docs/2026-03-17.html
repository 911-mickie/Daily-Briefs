• Extrinsic Hallucinations in LLMs — Lil'Log
Lilian Weng distinguishes extrinsic hallucination (output not grounded by world knowledge in pre-training data) from in-context hallucination (output contradicts the provided context). The post argues that extrinsic hallucination is fundamentally a retrieval problem: the model lacks the fact, and without RAG or tool access, it confabulates. Key causes: pre-training data noise/staleness, fine-tuning on synthetic data that introduces factual drift, and RLHF reward hacking toward fluency over factual accuracy.

Today's concept: Context Window Management in Multi-Agent Systems

Context Window Management in Multi-Agent Systems
The context window is the single most constrained resource in any agentic system — it determines what the model can "see" at decision time, and unlike compute, you can't simply throw more hardware at it. The core problem: as agents take more steps, accumulate tool outputs, and build up conversation history, the context fills with low-value tokens that crowd out high-value ones. Modern LLMs technically support up to 1M tokens, but quality benchmarks consistently degrade above 200k tokens, and cost scales linearly with tokens at inference time. The primary strategies are: (1) subagent dispatch — fresh context windows for bounded subtasks, as Claude Code does; (2) context compression — summarizing or truncating earlier turns; (3) selective retrieval — injecting only the top-k relevant chunks at each step rather than appending everything; and (4) structured state — maintaining a separate key-value state object outside the context that the agent reads selectively. A common misconception is that "bigger context window = better agent" — in practice, irrelevant context actively degrades reasoning quality (the "lost in the middle" problem), so aggressive context curation beats passive accumulation.

Q1: Why does a 1M token context window not simply eliminate context management as a problem? → Quality degrades and cost scales linearly well before the limit; "lost in the middle" means models under-attend to information not at the edges of context.
Q2: In Lexi, a user asks a question that requires synthesizing information from 50 files. How do you avoid burning the full context on file contents? → Use the summarization layer you already have — retrieve top-k file summaries first, then selectively fetch full content only for the highest-confidence files, using a two-stage retrieval pattern.
Q3: What's the tradeoff between context compression (summarizing history) vs. subagent dispatch (fresh context per task) for a long-running coding agent? → Compression loses precise details (exact variable names, stack traces) that matter for code tasks; subagents preserve precision but require careful handoff prompts to avoid losing critical state across boundaries.
