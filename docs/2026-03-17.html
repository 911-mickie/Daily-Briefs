💡 CONCEPT REFRESHER


Context Window Budget Management in Agentic Systems

In agentic LLM systems, the context window is a finite, non-replenishable resource per inference call — once full, you either truncate, summarize, or fail. Unlike single-turn generation, agents accumulate tool call results, conversation history, retrieved chunks, and intermediate reasoning traces across many steps, making context budget exhaustion the primary failure mode in long-horizon tasks. The core strategies are: (1) subagent delegation — dispatch scoped child agents with fresh windows for bounded subtasks and return only a compressed summary; (2) rolling summarization — periodically compress older context into a summary and evict raw history; (3) retrieval over memory — don't keep everything in context, externalize to a vector store and retrieve on demand. A common misconception is that larger context windows (1M tokens) eliminate this problem — benchmarks consistently show quality degradation well before the limit, and cost scales linearly with tokens used, making aggressive context trimming a production necessity even when headroom exists.

Q1: Why does Claude Code use a subagent for repo exploration rather than just loading all files into the main context? → To preserve top-level context budget for planning and synthesis, and because exploration is a bounded scoped task whose output can be summarized into a compact artifact.

Q2: You have a RAG pipeline where retrieved chunks plus chat history are approaching the context limit mid-session. What are your mitigation options in priority order? → First, compress/evict older chat turns via rolling summarization; second, reduce chunk count or chunk size from retrieval; third, re-rank retrieved chunks more aggressively to keep only the highest-signal ones.

Q3: In a multi-agent coding system, a worker subagent needs to return its findings to the orchestrator — what's the right output schema design to minimize orchestrator context consumption? → Return a structured, typed summary (e.g., JSON with file paths, relevance scores, and a one-sentence rationale per file) rather than raw content — the orchestrator consumes the schema, not the evidence, and fetches raw content on demand.
