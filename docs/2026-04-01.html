💡 CONCEPT REFRESHER

Today's concept: LoRA (Low-Rank Adaptation)

LoRA (Low-Rank Adaptation)
LoRA is a parameter-efficient fine-tuning technique that freezes pre-trained model weights and injects trainable low-rank decomposition matrices into specific layers — typically attention weight matrices — so that the weight update ΔW is approximated as a product of two low-rank matrices A and B (ΔW = BA, where rank r ≪ d). This reduces trainable parameters from millions to thousands while preserving the base model's general capabilities, enabling fine-tuning on consumer-grade hardware. In production, LoRA adapters can be swapped or merged at inference time, making them ideal for multi-task serving from a single base model — exactly the architecture Granite 4.0 3B Vision uses to keep vision and language modular. A common misconception is that LoRA always underperforms full fine-tuning; in practice, with the right rank and target modules, LoRA often matches or exceeds full fine-tuning quality at a fraction of the compute cost, particularly in data-limited regimes.

Why does LoRA decompose weight updates into two matrices A and B rather than directly learning ΔW?

Because constraining ΔW = BA where rank(B·A) = r forces the update to live in a low-dimensional subspace, drastically reducing trainable parameters while preserving expressive capacity for the target task.

How do you choose which layers to apply LoRA to, and what's the typical default?

Typically applied to query and value projection matrices in attention layers; including more layers (MLP, output projections) increases expressivity but also parameter count — the right choice depends on task complexity and compute budget.

If you have multiple LoRA adapters fine-tuned for different tasks, how do you serve them efficiently without running separate model replicas?

Merge adapters into the base model weights at load time for single-task serving, or use dynamic adapter loading (e.g., via PEFT's hot-swap) to share a single base model across requests — at the cost of adapter loading latency per switch.
