💡 CONCEPT REFRESHER

Precision, Recall, and F1 Score

Precision measures what fraction of your positive predictions are actually correct: TP / (TP + FP). Recall measures what fraction of actual positives your model correctly identifies: TP / (TP + FN). These two metrics are in tension — optimizing a retrieval threshold for recall tends to pull in more false positives, hurting precision, and vice versa. F1 score is their harmonic mean (2 × precision × recall / (precision + recall)), which penalizes extreme imbalances between the two more severely than a simple average. The most common misconception is treating F1 as a universal "best" metric — in production, whether you weight precision or recall depends entirely on the cost asymmetry of false positives vs. false negatives in your specific system.

In a medical diagnosis classifier, would you optimize for precision or recall, and why?

Recall — a false negative (missed diagnosis) is far more costly than a false positive (unnecessary follow-up), so you minimize FN even at the cost of more FP.

Why is F1 a harmonic mean rather than an arithmetic mean?

The harmonic mean is dominated by the smaller value, so a model that achieves 99% precision but 1% recall gets a near-zero F1 rather than a misleadingly moderate 50%.

In a RAG retrieval system, how do precision and recall map to retrieval quality, and what's the tradeoff when you increase the number of retrieved chunks?

Recall@k improves as you retrieve more chunks (fewer relevant docs missed), but precision@k degrades as more irrelevant chunks enter the context window, increasing noise for the generator and token cost.

• Behind the Scenes: How Database Traffic Control Works
What's new: PlanetScale's deep dive explains that Traffic Control is implemented as a Postgres extension that hooks into the planning and execution pipeline — intercepting queries post-parse, evaluating them against configured budget rules (query fingerprint, app name, Postgres user, SQL comment tags), and enforcing CPU%, concurrency, and per-query timing caps before execution proceeds. The extension uses Postgres's shared memory and background worker infrastructure to track cross-connection resource consumption in real time.
Practical takeaway: If you're building multi-tenant Postgres infrastructure, study this architecture — using SQL comment tags to propagate application-level metadata (feature name, customer tier, priority) into the query layer is a pattern you can implement today with pg_stat_statements and application-side conventions, even without Traffic Control.

• EXPLAIN's Other Superpowers
What's new: The post covers three underused EXPLAIN options: BUFFERS (shows shared hit vs. shared read per node, identifying cache miss hotspots at the 8KB block level), SETTINGS (surfaces non-default planner configuration active during planning), and GENERIC_PLAN (forces display of the generic plan used for parameterized queries, bypassing custom plan selection). The BUFFERS output distinguishes between shared buffers (RAM cache) and disk reads, making it the fastest way to confirm whether a performance problem is memory-bound or I/O-bound.
Practical takeaway: Add BUFFERS to every EXPLAIN ANALYZE you run on slow queries — if you're seeing high shared read counts on index scans, your shared_buffers or OS page cache is undersized for your working set.

• Waiting for PostgreSQL 19 – pg_plan_advice Contrib Module
What's new: The newly committed pg_plan_advice module lets you capture a finished query plan's key decisions as structured text, then inject that advice into future planning cycles to pin those decisions — for example, forcing a sequential scan on a specific relation or locking in a particular join order. Advice can be partial (controlling only some decisions), manually edited, or written from scratch, and is scoped to decisions like scan type and join strategy, not aggregation or sort order.
Practical takeaway: Use pg_plan_advice to stabilize critical query plans across statistics updates or minor version upgrades — it's a safer, more auditable alternative to embedding hints in application SQL or relying on pg_hint_plan.

• Introducing Database Traffic Control
What's new: Traffic Control introduces budget-based query throttling to Postgres, letting operators define resource limits per query pattern, application, user, or custom tag. Budgets enforce caps on CPU%, burst behavior, backend process concurrency, and per-query wall time, and can run in warn mode (observe only) before switching to enforce mode. This is the first production-grade Postgres-native solution to the "thundering herd of bad queries" problem that previously required external proxies or application-side circuit breakers.
Practical takeaway: Instrument your application queries with SQL comment tags (e.g., /* app=payments, priority=high */) now — this positions you to adopt budget-based throttling cleanly, whether via Traffic Control or any future equivalent.
