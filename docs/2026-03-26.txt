• <b><a href="https://www.microsoft.com/en-us/research/blog/systematic-debugging-for-ai-agents-introducing-the-agentrx-framework/">AgentRx: Systematic Debugging for AI Agents</a></b>
AgentRx addresses the core problem that agent failures in long-horizon, multi-agent trajectories are hard to localize because errors propagate silently across dozens of steps. The approach synthesizes "guarded, executable constraints" from tool schemas and domain policies, then logs step-by-step violations to identify the first unrecoverable failure point. On the AgentRx Benchmark — 115 manually annotated failed trajectories across three agent frameworks — it achieves +23.6% failure localization and +22.9% root-cause attribution over prompting baselines. The nine-category failure taxonomy alone is a useful reference for anyone instrumenting production agent systems.
<i>Interview angle:</i> How would you instrument a 50-step agent pipeline to detect and surface the earliest recoverable failure point without re-running the entire trajectory?

• <b><a href="https://www.microsoft.com/en-us/research/blog/from-raw-interaction-to-reusable-knowledge-rethinking-memory-for-ai-agents/">PlugMem: Transforming raw agent interactions into reusable knowledge</a></b>
PlugMem tackles the counterintuitive problem that giving agents more memory often degrades performance — raw interaction logs fill context windows with low-signal content, slowing retrieval and crowding out relevant information. Drawing on cognitive science distinctions between episodic memory, semantic facts, and procedural skills, PlugMem converts interaction history into structured, compact knowledge units rather than raw text chunks. A single plug-and-play module improved performance across diverse agent benchmarks while consuming fewer memory tokens. This is directly applicable to any RAG-adjacent memory layer in a long-running agent system.
<i>Interview angle:</i> What's the difference between storing raw conversation history vs. structured knowledge in an agent memory system, and when does each degrade retrieval quality?

• <b><a href="https://www.microsoft.com/en-us/research/blog/corpgen-advances-ai-agents-for-real-work/">CORPGEN: Multi-Horizon Task Environments for workplace AI agents</a></b>
CORPGEN identifies a critical blind spot in current agent benchmarks: they evaluate one task at a time, while real workplace productivity requires managing dozens of interdependent tasks simultaneously. The paper introduces Multi-Horizon Task Environments (MHTEs), where each task requires 10–30 dependent steps within a five-hour session. Under multi-task load, leading computer-using agents drop from 16.7% to 8.7% completion rates; the CORPGEN "digital employee" architecture — with hierarchical planning, memory isolation, and experiential learning — achieves up to 3.5x higher completion rates. Crucially, the gains come from system design, not model choice, making the patterns portable.
<i>Interview angle:</i> How would you architect memory isolation and task prioritization for an agent that must manage five concurrent, interdependent long-horizon tasks without context bleed?

Today's concept: Agentic Memory Architecture

<b>Agentic Memory Architecture</b>
Memory in AI agents refers to how an agent stores, retrieves, and applies past experience to current decisions. The dominant naive approach — appending raw interaction history to the context window — fails at scale because context fills with low-signal content, retrieval precision degrades, and inference cost balloons. The more principled approach, as demonstrated by PlugMem, mirrors cognitive science: separate episodic records (what happened) from semantic facts (what is true) and procedural knowledge (how to do things), storing each in a form optimized for its retrieval pattern. In production RAG and agentic systems, this translates to choosing between dense vector retrieval, structured key-value stores, and summarization pipelines depending on what type of memory is being accessed. A common misconception is that longer context windows solve the memory problem — they reduce the urgency, but retrieval precision and reasoning over noisy long contexts remain fundamentally hard.

<b>Q1:</b> Why does appending raw conversation history to an agent's context degrade performance over time? → <i>Low-signal historical content crowds out relevant context, increases retrieval noise, and fills the context window, reducing the agent's effective reasoning space.</i>
<b>Q2:</b> What is the difference between episodic and semantic memory in the context of an LLM agent, and how would you store each? → <i>Episodic memory is event-specific (raw logs, past dialogues) best stored in a vector DB for similarity retrieval; semantic memory is distilled facts best stored in a structured KV store or summary index for exact or keyword lookup.</i>
<b>Q3:</b> In a multi-task agent managing five concurrent long-horizon tasks, how would you prevent memory bleed between tasks while still enabling cross-task learning? → <i>Use per-task memory isolation with separate namespaced retrieval indices, and periodically distill cross-task generalizations into a shared procedural memory layer that is read-only during task execution.</i>
