💡 CONCEPT REFRESHER

Today's concept: Tensor Parallelism vs. Pipeline Parallelism in LLM Serving

Tensor Parallelism vs. Pipeline Parallelism
Tensor parallelism (TP) shards individual weight matrices across GPUs — each GPU holds a slice of every layer and participates in every forward pass via AllReduce collectives, keeping latency low at the cost of high inter-GPU bandwidth per token. Pipeline parallelism (PP) assigns different layers to different GPUs, passing activations sequentially — this reduces per-device memory and communication bandwidth but introduces pipeline bubbles that hurt latency unless carefully scheduled with microbatching. In practice, production systems like Meta's (covered today) combine both: TP within a node (fast NVLink) and PP across nodes (slower interconnects), with context parallelism layered on top for long sequences. A common misconception is that more parallelism always improves throughput — in the decoding stage, which is memory-bandwidth-bound, adding TP beyond a certain point increases synchronization overhead faster than it reduces memory pressure per device. Expert parallelism in MoE models adds a third dimension: routing tokens to expert-specific GPUs via AllToAll, which has entirely different communication patterns than dense layer sharding.

Why is tensor parallelism preferred within a single node rather than across nodes?

NVLink bandwidth (~600 GB/s) can sustain the AllReduce overhead TP requires; PCIe/InfiniBand across nodes cannot.

Why does adding more tensor-parallel GPUs during decoding eventually hurt rather than help throughput?

Decoding is memory-bandwidth-bound; beyond the point where weights fit comfortably, AllReduce synchronization latency starts dominating over memory read savings.

In an MoE model served with expert parallelism, what new failure mode appears that doesn't exist in dense TP?

Load imbalance — if routing collapses to a subset of experts, some GPUs sit idle while others are overloaded, degrading both throughput and latency unpredictably.
